## Project background

**Project**: AutoFusion-Bench  
**Domain**: budget-constrained multimodal fusion decision benchmarking  
**Current formalism version**: v1

AutoFusion-Bench is testing whether multimodal fusion should be evaluated as a
decision problem over reliability and inference-budget conditions, rather than
as one static architecture score. The current pilot is deliberately small: a
frozen 7-template tri-modal registry over a 2x2 reliability-budget surface. The
formalism file is still skeletal, so the experiment contract below is the main
source of truth.

## Experiment setup

- **Dataset**: MELD first.
- **Hardware target**: `ntu-gpu43`, direct SSH/tmux execution, 4x RTX A5000
  available but current producers are CPU/OpenCV/sklearn feature-level runs.
- **Feature-level boundary**: this pilot does not train end-to-end MELD encoders
  and must not be framed as end-to-end deployment latency evidence.
- **Templates**: `T`, `A`, `V`, `TA`, `TV`, `AV`, `TAV`.
- **Conditions**: `clean_loose`, `clean_tight`, `degraded_loose`,
  `degraded_tight`.
- **Degraded slices**: `degraded_text`, `degraded_audio`, `degraded_video`,
  `mixed_degraded`.
- **Policies**: `random_legal`, `clean_best`, `static_full`, `budget_only`,
  `reliability_only`, `joint`, `feasible_oracle`.
- **Seeds**: 3.
- **Primary metric**: `best_single_axis_oracle_regret_dt`.
- **Success threshold**: primary regret >= 3 macro-F1 points and >= 1.5x pooled
  SE, or Kendall tau-b <= 0.3 as a secondary benchmark-signal route.
- **Joint policy success**: `joint_gap_closure >= 30%` and
  `post_mask_budget_violation_rate <= 1%`.

The main scientific question is whether the benchmark surface exposes a real
decision gap under `degraded_tight`, not whether the current joint policy is a
good new method.

The official MELD feature tarball contained text/audio/bimodal pickles and model
weights, but no visual feature pickles. We therefore tested two visual producer
paths:

1. `raw_stats`: file-size/path/timing-style raw MP4 statistics as weak visual
   features.
2. `cv2_stats`: OpenCV-decoded frame statistics including color, HSV, texture,
   histogram, motion, and metadata features. One corrupt MP4 was tolerated by
   fallback to padded raw-stat features.

For audio, the improved run used `official_concat`, concatenating the official
utterance-level audio embeddings with official dialogue-sequence audio features
when coverage was complete.

Each real producer generated the six required runner tables:

## What I expected vs what I see

Expected:

- In `degraded_tight`, the feasible oracle should beat the best single-axis
  policy by at least 3 macro-F1 points.
- Degraded modality-specific slices should create enough template ranking
  inversion that a reliability-budget decision surface is visible.
- Clean-only/static-full/budget-only/reliability-only policies should expose
  different blind spots.

Observed:

- Both full producers pass protocol, leakage, q-diagnostic, corruption, and
  post-mask legality gates.
- Both full producers fail benchmark-signal criteria.
- The best single-axis policy remains close to feasible oracle in
  `degraded_tight`.
- `reliability_only` has meaningful illegal proposal behavior in the cv2 run
  (`pre_mask_illegal_proposal_rate=0.500`) and performs poorly under tight, but
  budget-only is already close to oracle.
- The task appears strongly text-dominant: text-heavy policies remain hard to
  beat even under degraded conditions.

## Hypotheses for the gap

- H1: MELD with the current official text/audio features is too text-dominant
  for this minimal feature-level decision-surface pilot.
- H2: `cv2_stats` is still not semantically strong enough; it adds measurable
  budget spread but not useful visual discrimination.
- H3: The modality-specific degradation design does not sufficiently change the
  relative utility of templates at full scale.
- H4: The policy/oracle metric is too coarse because the feasible oracle and
  budget-only both exploit the same validation utility table, making the oracle
  gap small unless template rankings invert strongly.
- H5: The 7-template feature-head registry is sufficient as a first registry,
  but current features/corruptions do not create a hard enough selection problem.
- H6: The 3-F1 threshold may be appropriate for a paper-level positive result,
  but too strict for this first MELD feature-level producer.

## Constraints for your advice

- Do not recommend claiming the current cv2 run as a positive result.
- Keep the claim boundary feature-level unless proposing a concrete

- Preserve the main benchmark question: whether a reliability-budget surface
