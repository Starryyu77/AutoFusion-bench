# exp-001 memory

Updated: 2026-05-14

## Current objective

Implement and validate the `exp-001` decision-surface pilot execution scaffold:

- frozen 7-template registry
- feature-level cost/budget gate
- policy evaluation for `random_legal`, `clean_best`, `static_full`,
  `budget_only`, `reliability_only`, `joint`, and `feasible_oracle`
- primary metric `best_single_axis_oracle_regret_dt`
- leakage and corruption-manifest checks
- local fixture smoke that validates protocol code but is not an empirical result
- full MELD feature-level producer runs on `ntu-gpu43`

## Boundary

The current implementation is protocol/analysis infrastructure plus
feature-level MELD table producers. It does not train end-to-end MELD encoders
and does not support an end-to-end deployment-latency claim.

Fixture outputs and small-sample smoke outputs must not be cited as benchmark
evidence. The full `raw_stats` and full `cv2_stats` MELD runs are real
feature-level empirical runs, but both are benchmark-signal negative.

## External results review on 2026-05-14

- Incoming review saved at:

- Decision recorded at:
  `decisions/2026-05-14-exp001-semantic-visual-rerun.md`
- Review conclusion:
  - Do not claim the current MELD `cv2_stats` run as a positive result.
  - Do not stop `exp-001`.
  - Treat current `cv2_stats` as `negative-but-diagnostic`, not protocol
    failure.
  - Keep the 3 macro-F1 positive threshold and existing primary metric.
  - Do not switch dataset, harden corruption, or change metric before the next
    controlled producer test.
- Accepted next route:
  - working label: `exp-001-meld-semvis-v1`
  - keep MELD splits, 7 templates, 2x2 reliability-budget surface, policies,
    seeds, degradation manifest, and success criteria unchanged
  - replace only `video_source=cv2_stats` with a frozen semantic visual
    embedding producer
  - use frame-level frozen CLIP/ViT or VideoMAE-style embeddings with
    utterance-level mean pooling
  - no supervised visual fine-tuning and no architecture search
- Interpretation rule:
  - if semantic visual regret reaches `>=3.0` and rank inversion improves,
    exp-001 has a positive MELD feature-level result
  - if regret is `1.0-3.0` with improved rank inversion, keep MELD as diagnostic
    but do not claim positive benchmark signal
  - if regret remains `<1.0` and text remains dominant, stop using MELD as the
    primary benchmark-signal dataset for exp-001
  - if semantic visual improves visual templates but `budget_only` still tracks
    oracle closely, inspect whether the primary metric needs a paired
    diagnostic rather than replacing it

## Semantic visual producer implementation on 2026-05-14

- Added `video_source=semvis_clip` to the MELD table producer.
- Default frozen visual encoder:
  `openai/clip-vit-base-patch32`
- The producer:
  - samples `--semvis-frame-count` frames per utterance video, default `8`
  - encodes frames with frozen CLIP image embeddings
  - L2-normalizes frame embeddings
  - mean-pools them into one utterance-level visual vector
  - caches vectors under the producer output `feature-cache`
  - tolerates corrupt/non-decodable MP4 files by assigning a zero semantic
    visual vector for that utterance
- CLI route:
  `python3 -m autofusion_bench.exp001.run_meld_table_producer --video-source semvis_clip`
- New CLI controls:
  - `--semvis-model`
  - `--semvis-frame-count`
  - `--semvis-batch-frames`
  - `--semvis-device`
- Keep the claim boundary: this is a frozen feature-level semantic visual
  producer. The cost table remains cached-feature template-head decision cost,
  not end-to-end deployment latency.
